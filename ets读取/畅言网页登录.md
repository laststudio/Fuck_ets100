# ETS100 畅言通行证网页登录开发文档

> 本文档描述 ETS100 `user/login-zt` 网页登录链路。该链路用于在 Android WebView 中打开畅言通行证页面，通过网页登录态换取 ETS100 token，再进入云端作业读取流程。

[API 开发文档](./API_DOC.md) | [云端模式逻辑说明](./云端模式逻辑文档.md)

---

# 一、整体目标与运行模型

畅言网页登录的核心目标：

```text
打开畅言网页登录页
    ↓
用户完成通行证登录
    ↓
拦截 /login/checkLogin 响应
    ↓
提取 uid + captchaResult
    ↓
调用 /user/login-zt 换取 ETS100 token
    ↓
调用 /m/ecard/list 获取可用账号列表
    ↓
选择有效账号并保存 parent_account_id
    ↓
使用 /g/homework/list 读取云端作业
```

## 适用场景

| 场景 | 说明 |
|------|------|
| Android WebView | 当前 Fe 项目的实现方式 |
| 企业/通行证账号 | 适用于无法直接使用 `/user/login` 的通行证网页登录 |
| 多年级账号 | 需要从 `/m/ecard/list` 中选择正确可用账号 |

## 与手机号密码登录的区别

| 对比项 | 手机号密码登录 | 畅言网页登录 |
|--------|----------------|--------------|
| 登录入口 | App 原生输入框 | 独立 WebView Activity |
| 登录接口 | `/user/login` | `/user/login-zt` |
| 凭据来源 | 手机号 + 密码 | `checkLogin` 的 `uid` + `captchaResult` |
| 登录名 | 用户输入手机号 | 当前实测应使用 `uid` |
| 多账号处理 | 通常取父账户列表 | 必须筛选有效账号，必要时让用户选择 |

# 三、完整工作流程

```
Step 1: 用户进入云端登录页
    │
    ├─ 原有方式：手机号 + 密码
    └─ 新方式：点击小字按钮“讯飞登录”
        │
        ▼
Step 2: 启动 ChangyanWebLoginActivity
    │  创建 WebView
    │  设置手机 WebView UA
    │  加载 pass.changyan.com 登录页
    │
    ▼
Step 3: 页面加载完成后注入 JS
    │  监听 jQuery ajaxComplete
    │  只处理 /login/checkLogin
    │
    ▼
Step 4: 用户完成网页登录
    │  畅言页面发起 /login/checkLogin
    │  JS 把响应传给 Android JavascriptInterface
    │
    ▼
Step 5: Android 解析 checkLogin
    │  HTTP status 必须为 200
    │  responseBody.Code 必须为 0
    │  responseBody.Data 需要二次 JSON 解析
    │
    ▼
Step 6: 调用 /user/login-zt
    │  login = uid
    │  user_id = uid
    │  suc_user_token = captchaResult
    │
    ▼
Step 7: 调用 /m/ecard/list
    │  获取该 token 下全部账号/年级/班级条目
    │
    ▼
Step 8: 选择有效账号
    │  如果只有 1 个有效账号：自动选择
    │  如果多个有效账号：弹窗让用户选择
    │  保存 selected_ecard_id 与 parent_account_id
    │
    ▼
Step 9: 回到云端模式
    │  后续 /g/homework/list 使用保存的 parent_account_id
```

# 五、JS 拦截 checkLogin

## 5.1 注入时机

在 `WebViewClient.onPageFinished()` 中注入。页面每次跳转完成都可以尝试注入，但要通过 `window.__feEtsAjaxCaptureInstalled` 避免重复绑定。

## 5.2 拦截逻辑

```javascript
(function() {
    if (window.__feEtsAjaxCaptureInstalled) return;
    window.__feEtsAjaxCaptureInstalled = true;
    if (!window.$) return;

    $(document).ajaxComplete(function(event, xhr, settings) {
        if (!settings || !settings.url || settings.url.indexOf('/login/checkLogin') === -1) return;

        var requestParams = {};
        if (settings.data) {
            if (typeof settings.data === 'string') {
                try {
                    requestParams = Object.fromEntries(new URLSearchParams(settings.data));
                } catch(e) {}
            } else {
                requestParams = settings.data;
            }
        }

        window.etsBridge.capture(JSON.stringify({
            url: settings.url,
            method: settings.type || 'GET',
            requestParams: requestParams,
            status: xhr.status,
            responseBody: xhr.responseText || ''
        }));
    });
})();
```

# 六、checkLogin 响应解析

## 6.1 响应示例

```json
{
  "Code": 0,
  "Data": "{\"uid\":\"1500000100220922169\",\"captchaResult\":\"9357c03b-13d4-443a-9097-3dcc82e6ecc9\"}"
}
```

## 6.2 字段映射

| 来源 | 字段 | 当前用途 |
|------|------|----------|
| `responseBody.Data.uid` | `uid` | 同时作为 `/user/login-zt` 的 `login` 与 `user_id` |
| `responseBody.Data.captchaResult` | `captchaResult` | `/user/login-zt` 的 `suc_user_token` |
| `requestParams.i` | 旧登录名 | 实测企业账号下不可靠，不再作为 `login` |

## 6.3 已踩坑

| 问题 | 现象 | 处理 |
|------|------|------|
| 使用 `requestParams.i` 作为登录名 | token 可拿到但后续账号链路不稳定 | 当前使用 `uid` 作为 `login` |
| 只看 WebView 系统日志 | 只能看到 `VRI[Changy...Activity]` | 过滤 `ChangyanWebLogin` 与 `ETS100ApiClient` |
| JS 只在第一页注入 | 登录后跳转页可能没有拦截器 | 每次 `onPageFinished` 都尝试注入 |

# 八、账号选择与 parent_account_id

## 8.1 `/m/ecard/list` 作用

`/m/ecard/list` 返回当前 token 下所有可用/过期账号条目。后续 `/g/homework/list` 需要的是所选条目的 `parent_id`，即 `parent_account_id`。

## 8.2 关键字段

| 字段 | 说明 |
|------|------|
| `id` | ecard 条目 ID，可用于保存用户选择 |
| `parent_id` | 作业列表接口使用的 `parent_account_id` |
| `user_account_id` | 子账户 ID，当前不直接传给作业接口 |
| `name` / `grade` | 展示给用户选择 |
| `status` | `0` 表示可用，`3` 实测为过期或不可用 |
| `out_of_date` | `0` 表示未过期 |
| `class_id` | 有效班级通常非空 |
| `mobile_status` | 移动端可用状态 |

## 8.3 已踩坑：不能固定取第一个账号

实测账号返回：

| 条目 | 年级 | 状态 | parent_id | 结论 |
|------|------|------|-----------|------|
| `ecard[0]` | 高一 | `status=3`, `out_of_date=1`, `class_id=` | `150472378` | 过期账号，传给作业接口会返回 `80004` |
| `ecard[1]` | 高二 | `status=3`, `out_of_date=1`, `class_id=` | `150472378` | 过期账号 |
| `ecard[3]` | 七年级 | `status=0`, `out_of_date=0`, `class_id=531216` | `150472374` | 有效账号 |
| `ecard[4]` | 八年级 | `status=0`, `out_of_date=0`, `class_id=531216` | `150472374` | 有效账号 |

如果固定取第一个账号，会保存 `parent_account_id=150472378`，随后 `/g/homework/list` 返回：

```json
[{ "msg": "没有找到有效班级", "code": 80004 }]
```

## 8.4 已踩坑：不能硬编码八年级

硬编码 `grade == "八年级"` 可以修复特定测试账号，但其他用户可能是七年级、九年级或高中账号。通用实现应：

1. 过滤有效账号：
   - `status == 0`
   - `out_of_date == 0`
   - `class_id` 非空
   - `mobile_status == 1`
2. 如果只有一个有效账号，自动选择；
3. 如果多个有效账号，弹窗让用户选择；
4. 保存所选 `ecard.id`，下次优先复用；
5. 如果保存的 `ecard.id` 失效，再重新弹窗选择。

## 8.5 推荐持久化字段

| 字段 | 用途 |
|------|------|
| `selected_ecard_id` | 用户选择的 ecard 条目 |
| `parent_account_id` | 作业列表请求参数 |
| `selected_ecard_name` | 展示用，如 `北京八年级` |
| `selected_ecard_grade` | 展示用，如 `八年级` |
| `selected_ecard_class_id` | 诊断与有效性判断 |

# 十、实现注意事项

1. **不要固定取 `ecard[0]`**：第一个条目可能是过期账号。
2. **不要硬编码年级**：不同用户年级不同，应让用户选择。
3. **不要盲信 `recent_account_id`**：它可能指向最近使用账号，但不一定是用户当前想读取的年级。
4. **Token 成功不代表作业接口可用**：`token` 有效但 `parent_account_id` 错误时，`/g/homework/list` 仍会返回 `80004`。
5. **敏感日志要掩码**：token、captchaResult、deviceCode 只打印首尾片段。
6. **WebView 日志和系统窗口日志不同**：`VRI[Changy...]` 是系统窗口日志，业务日志 tag 是 `ChangyanWebLogin`。
